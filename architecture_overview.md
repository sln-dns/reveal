# Architecture Overview

Этот файл описывает высокоуровневую архитектуру продукта как веб-приложения с асинхронной парной механикой, сценарным движком и LLM-генерацией.

## Core architecture idea

Продукт стоит мыслить не как обычный чат или опросник, а как `stateful multiplayer scenario app with guided generation`.

Центр системы:

- не сообщения сами по себе;
- не просто список вопросов;
- а состояние конкретной пары внутри конкретного сценарного прохождения.

Главное архитектурное разделение:

1. Есть структура сценария.
2. Есть конкретный запуск сценария для пары.
3. Есть сгенерированный LLM-контент для этого запуска.

## Main layers

### 1. Web client

Пользовательский интерфейс, через который игроки проходят сценарий.

Основные задачи:

- вход по ссылке в сессию;
- подключение к уже созданной сессии;
- просмотр текущей сцены и текущего вопроса;
- отправка ответа;
- ожидание второго игрока;
- одновременное раскрытие ответов;
- движение по сценам;
- пауза и возврат в сценарий;
- просмотр финального summary.

Ключевые UX-требования:

- очень низкое трение на входе;
- понятный статус "ты ответил / ждём второго";
- ощущение прогресса;
- хороший опыт асинхронного возвращения;
- ясный финал с мостом в продолжение общения.

### 2. App backend

Основной сервер приложения, который оркестрирует продуктовую и техническую логику.

Основные задачи:

- создание сессий;
- присоединение участников;
- управление доступом;
- приём ответов;
- проверка условий reveal;
- переключение шагов и сцен;
- вызов сценарного движка;
- вызов LLM-слоя;
- сохранение состояния;
- выдача текущего состояния клиенту.

Этот слой должен быть координатором, но не местом, где размазана вся сценарная логика.

### 3. Scenario engine

Отдельный логический слой, который отвечает за исполнение сценария.

Основные задачи:

- загрузка и валидация сценарного blueprint JSON;
- понимание текущей сцены и текущей позиции пары;
- применение правил ветвления;
- определение допустимых question families;
- контроль психологической ступени;
- сбор payload для LLM;
- расчёт следующего шага после двух ответов;
- подготовка summary context в финале.

Именно этот слой превращает сценарий из "документа" в работающий движок.

### 4. LLM generation layer

Слой, который общается с моделью и отвечает за генерацию живого контента.

Основные задачи:

- генерация мира и атмосферы в рамках blueprint;
- генерация вводного текста сцены;
- генерация конкретных вопросов;
- генерация переходов между сценами;
- генерация финального summary для каждого игрока.

Принцип работы:

- LLM не должна сама принимать ключевые продуктовые решения;
- LLM получает структурированный payload от scenario engine;
- LLM возвращает текстовый и контентный слой, но не ломает рамки сценария.

### 5. Persistence layer

Слой хранения данных.

Он должен хранить не только пользователей, но и живое состояние сценарных прохождений.

На уровне сущностей логично ожидать:

- `user`
- `session`
- `session_participant`
- `scenario_blueprint`
- `scenario_run`
- `scene_instance`
- `question_instance`
- `answer`
- `reveal_event`
- `summary`
- `notification_event`
- `generation_log`

Важно разделять:

- blueprint-уровень;
- runtime-уровень;
- generated-content уровень.

### 6. Notification layer

Так как опыт асинхронный, возвращение пользователя в сценарий является частью core loop.

Основные задачи:

- уведомить второго игрока, что пришёл новый вопрос;
- уведомить первого игрока, что ответы раскрылись;
- уведомить обоих, что доступен следующий шаг;
- отправлять мягкие reminders, если один из игроков задержался;
- отправлять финальные уведомления о summary или завершении маршрута.

На раннем этапе это может быть простой in-app и e-mail слой, позже можно добавить push, Telegram и другие каналы.

### 7. Logging and observability layer

Это обязательный слой, а не второстепенный.

Для такого продукта логгирование нужно не только для технических ошибок, но и для понимания качества сценария, поведения LLM и продуктового воронки.

Основные задачи:

- логгировать ключевые продуктовые события;
- логгировать переходы между сценами;
- логгировать reveal-события;
- логгировать LLM-вызовы и их результаты;
- логгировать ошибки и деградации сценарного рантайма;
- собирать метрики по вовлечению, completion и зависаниям.

Что важно логгировать:

- создание сессии;
- присоединение второго игрока;
- показ сцены;
- показ вопроса;
- отправку ответа;
- момент reveal;
- выбор ветки;
- генерацию сцены моделью;
- генерацию summary;
- напоминания и возвраты;
- завершение сценария;
- точки выхода пользователей.

Что важно измерять:

- invite acceptance rate;
- answer completion rate;
- reveal rate;
- progression from scene to scene;
- median waiting time for second answer;
- final completion rate;
- summary open rate;
- повторные запуски новых сессий тем же пользователем.

Отдельно стоит логгировать LLM-качество:

- какой blueprint и какая сцена были на входе;
- какой prompt payload был собран;
- какой ответ вернула модель;
- были ли пост-валидационные ошибки;
- пришлось ли делать retry или fallback;
- были ли нарушения формата или tone constraints.

## Data model levels

### 1. Blueprint level

Это данные сценария как шаблона.

Примеры:

- JSON blueprint;
- scene definitions;
- ladder stages;
- branching rules;
- question family rules.

### 2. Run level

Это конкретное прохождение сценария конкретной парой.

Примеры:

- кто участвует;
- какой выбран мир;
- какой выбран тон;
- на какой сцене находится пара;
- какие ответы уже даны;
- был ли reveal;
- какая ветка выбрана;
- paused или active run.

### 3. Generated content level

Это то, что было создано моделью для конкретного запуска.

Примеры:

- текст вступления;
- сгенерированные вопросы;
- переходы;
- generated summary;
- служебные артефакты генерации.

## Runtime flow

Ниже базовый поток работы системы.

1. Один пользователь создаёт сессию.
2. Второй пользователь присоединяется по ссылке.
3. Backend создаёт `scenario_run`.
4. Scenario engine загружает blueprint и выбирает стартовые параметры.
5. LLM layer генерирует первую сцену и первые вопросы.
6. Оба пользователя получают текущий шаг.
7. Один пользователь отвечает.
8. Ответ сохраняется, но reveal не происходит.
9. Второй пользователь отвечает.
10. Backend открывает ответы одновременно.
11. Scenario engine анализирует контекст и выбирает следующую сцену или следующий шаг.
12. При необходимости LLM генерирует следующий контент.
13. Цикл повторяется до финала.
14. В конце генерируется отдельное summary для каждого игрока.

## Suggested code structure

На уровне кода проект логично делить на следующие модули:

- `web-app`
- `api`
- `scenario-engine`
- `llm-service`
- `notifications`
- `persistence`
- `logging-observability`
- `shared-types`

### `web-app`

UI-компоненты, клиентское состояние, переходы между экранами, обработка асинхронного UX.

### `api`

HTTP или RPC слой, который связывает фронтенд и серверную логику.

### `scenario-engine`

Выполнение blueprint, выбор сцен, валидация переходов, сбор prompt payload.

### `llm-service`

Промпты, вызовы модели, валидация ответов модели, fallback-логика.

### `notifications`

Планирование и отправка уведомлений.

### `persistence`

Работа с базой, репозитории, хранение state и generated content.

### `logging-observability`

Событийные логи, технические логи, метрики, трассировка LLM и сценарных переходов.

### `shared-types`

Общие контракты и типы для blueprint, runtime state, generation payload и API.

## API groups

На высоком уровне можно ожидать такие группы endpoint'ов:

- session management

## Architectural principles

### Structure over freeform generation

Сценарий должен быть управляемым. Генерация идёт внутри рамок, а не вместо рамок.

### State first

Главный источник истины - не клиент и не LLM, а состояние `scenario_run`.

### Deterministic flow, generative presentation

Маршрут и правила переходов должны быть детерминированными, а текст, мир и подача - генеративными.

### Async-first design

Асинхронность не побочная функция, а центральная часть продукта.

### Observability by design

С самого начала нужно проектировать систему так, чтобы было видно:

- где ломается UX;
- где зависают пары;
- где плохо срабатывает генерация;
- где рушится progression;
- где LLM уходит не в тот тон или уровень глубины.

## Main risks to keep in mind

- Смешать сценарную логику с API-кодом.
- Отдать слишком много контроля LLM.
- Недооценить сложность асинхронного UX.
- Не разделить blueprint, run state и generated content.
- Не собрать достаточное логгирование и потом не понимать, где ломается сценарий.

## Near-term implementation strategy

На раннем этапе разумно строить систему в таком порядке:

1. Реализовать минимальный scenario engine.
2. Реализовать session/run state.
3. Реализовать базовый клиентский flow.
4. Подключить LLM только для генерации текстового слоя.
5. Добавить событийное логгирование.
6. Только потом усложнять ветвления, миры и summary quality.

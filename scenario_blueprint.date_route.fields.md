# Scenario Blueprint Fields

Этот файл описывает поля из [scenario_blueprint.date_route.json](/Users/denis/projects/Idea_check/scenario_blueprint.date_route.json) и нужен как справочник для дальнейшей разработки каркаса сценария.

## Top level

### Field roles

Для этого JSON полезно думать о трёх ролях полей:

- `engine required`
  Поля, без которых рантайм не сможет корректно провести сценарий.
- `prompt required`
  Поля, которые обязательно стоит передавать LLM, чтобы она генерировала мир, сцены и вопросы не хаотично, а в рамках сценария.
- `editorial optional`
  Поля, которые удобны для людей, редактирования и документации, но не критичны для исполнения.

### `schema_version`

Версия схемы JSON-каркаса. Нужна, чтобы позже можно было безопасно развивать формат и не ломать старые сценарии.

### `scenario_id`

Уникальный технический идентификатор сценария.

### `scenario_name`

Человекочитаемое название сценария.

### `scenario_type`

Категория сценария. Помогает различать разные виды сценариев, если позже появятся другие режимы кроме `date_route`.

### `status`

Стадия готовности сценария. Например: `draft`, `testing`, `active`.

### `description`

Короткое описание назначения сценария.

### `product_goal`

Главная продуктовая цель сценария. Это поле полезно и для команды, и как часть контекста для промпта модели.

### `generation_architecture`

Служебный раздел, который явно разделяет секции JSON по ролям:

- что обязательно для движка;
- что обязательно для промпта LLM;
- что остаётся редакторским удобством.

Это полезно, чтобы не пытаться скормить модели весь JSON целиком без разбора.

### `experience_principles`

Список принципов, которые сценарий должен сохранять. Это высокоуровневые ограничения на UX и tone of experience.

## `session_model`

Описывает базовые правила прохождения сессии.

### `players_count`

Сколько игроков участвуют в сценарии.

### `mode`

Тип взаимодействия. В текущей идее это `asynchronous`.

### `answer_reveal`

Правило раскрытия ответов. Значение `both_answers_required` означает, что ответы открываются только после двух ответов.

### `pause_resume_allowed`

Можно ли прерывать и потом продолжать прохождение.

### `estimated_total_steps`

Примерная оценка общего числа шагов в сценарии.

### `estimated_scene_count`

Примерная оценка числа сцен.

## `world_setup`

Описывает, как задаётся мир, атмосфера и общая рамка прохождения.

### `selection_mode`

Как выбирается мир. Сейчас это случайный выбор из заранее заданных пресетов.

### `preset_world_ids`

Список доступных пресетов миров для MVP.

### `tone_mode`

Как выбирается тон. В текущем черновике он меняется от запуска к запуску.

### `allowed_tones`

Список тональностей, которые допустимы для сценария.

### `realism_mode`

Правило реалистичности мира. Значение `player_inferred` означает, что мир может быть как реалистичным, так и фантазийным, а допустимость этого считывается из сценарного контекста.

## `progression`

Описывает, как ощущается продвижение по игре.

### `unit`

Основная единица прогресса. Сейчас это `scene`.

### `show_progress_to_users`

Показывать ли пользователю явный индикатор прогресса.

### `progress_label_template`

Шаблон подписи прогресса.

### `interruption_safe`

Можно ли безопасно прерывать прохождение между шагами и сценами.

## `scene_types`

Справочный список допустимых типов сцен в сценарии. Эти типы нужны, чтобы не строить маршрут полностью хаотично.

## `question_policy`

Описывает правила для вопросов и ответов.

### `default_answer_format`

Основной формат ответа для MVP.

### `allowed_answer_formats`

Список допустимых форматов ответа.

### `preferred_question_style`

Текущий продуктовый принцип для генерации вопросов. В `date_route` это смещение к формату быстрого выбора из нескольких коротких вариантов с возможностью коротко написать свой ответ.

### `preferred_option_count_min` / `preferred_option_count_max`

Желаемый диапазон числа вариантов внутри одного вопроса. Эти поля нужны prompt builder и будущему UI, чтобы вопрос не расползался в длинное перечисление.

### `allow_custom_answer_option`

Флаг, который закрепляет правило: даже если основной режим вопроса это выбор из вариантов, у пользователя должна оставаться возможность ответить своим коротким текстом.

### `custom_answer_label`

Подпись для свободного варианта ответа. Сейчас продукт использует формулировку `свой вариант`.

### `generation_rules`

Редакторский и продуктовый список правил, которые удерживают генерацию в low-friction формате: быстрый выбор, минимум рефлексии, короткие варианты, живая атмосфера без тестового тона.

### `questions_per_scene_min`

Минимум вопросов внутри одной сцены.

### `questions_per_scene_max`

Максимум вопросов внутри одной сцены.

### `max_answer_length_chars`

Ограничение на длину ответа, чтобы поддерживать лёгкость опыта.

### `llm_interpretation_during_run`

Насколько активно модель интерпретирует ответы во время прохождения. Значение `minimal` означает, что система не должна делать заметные психологические выводы по ходу игры.

### `direct_psychological_analysis`

Разрешено ли модели делать прямой психологический анализ. Для MVP это запрещено.

## `branching_policy`

Описывает правила ветвления между сценами.

### `difference_strategy`

Что делать, если ответы игроков различаются. Значение `soft_compromise` означает мягкий переход к компромиссной ветке.

### `match_strategy`

Что делать, если ответы совпали.

### `hard_blocking_on_difference`

Можно ли стопорить прохождение из-за различий. Для MVP это выключено.

### `fallback_branch_type`

Какой тип ветки использовать, если нужно обойти различие.

### `branching_inputs`

Какие сигналы могут использоваться для выбора следующей ветки.

## `summary_policy`

Описывает логику финального summary.

### `generate_summary`

Нужно ли вообще генерировать summary.

### `summary_per_player`

Получает ли каждый игрок отдельное summary про другого.

### `summary_focus`

Какие аспекты должны быть отражены в summary.

### `summary_tone`

Тон summary.

### `forbidden_summary_styles`

Форматы и стили, которых summary должно избегать.

## `scene_flow`

Главный массив сцен. Каждая сцена описывает один крупный этап маршрута.

### Поля объекта сцены

Особенно важное поле здесь это `question_templates`: теперь оно должно задавать не просто тему вопроса, а желаемую форму вопроса. Для текущего продукта это преимущественно choice-style формулировки вида `вариант А, вариант Б, вариант В или свой вариант?`, а не открытые вопросы на длинное размышление.

#### `scene_id`

Уникальный идентификатор сцены.

#### `scene_type`

Тип сцены из списка `scene_types`.

#### `title`

Внутреннее название сцены.

#### `purpose`

Зачем сцена нужна с продуктовой точки зрения. Это одно из важнейших полей: оно задаёт функцию сцены, а не её художественный текст.

#### `ladder_stages`

Какие ступени психологической лестницы обслуживает сцена. Это важное поле для промпта, потому что помогает модели не перескакивать по глубине слишком резко.

#### `psychological_goal`

Короткое описание психологической задачи сцены. Это один из лучших сигналов для LLM: он объясняет не "о чём текст", а "что должно произойти с пользователем".

#### `allowed_question_families`

Какие семейства вопросов допустимы внутри этой сцены. Помогает держать качество генерации и не получать вопросы "не того уровня".

#### `forbidden_question_families`

Какие типы вопросов в этой сцене запрещены. Это так же важно для промпта, как и разрешённые типы.

#### `question_count_target`

Желаемое количество вопросов в сцене.

#### `entry_text_mode`

Как формируется вводный текст сцены. Например: полностью генерируется моделью или собирается по шаблону.

#### `question_templates`

Черновые шаблоны вопросов или смысловые опоры для генерации. Это не обязательно финальные реплики для пользователя.

#### `transition_goal`

Что пользователь должен унести из сцены в следующую сцену. Это помогает модели писать переходы и завершать сцену с нужным эффектом.

#### `branch_outcomes`

Описание того, куда маршрут должен идти после этой сцены.

### Особые поля внутри `branch_outcomes`

#### `default_next_scene_id`

Следующая сцена по умолчанию.

#### `if_match`

Следующая сцена, если ответы игроков в этой точке достаточно совпали.

#### `if_difference`

Следующая сцена, если ответы игроков заметно различаются.

#### `end_scenario`

Флаг завершения сценария.

## `llm_generation_contract`

Контракт между структурой сценария и моделью.

### `input_sections`

Какие части JSON и контекста мы подаём в промпт модели при генерации конкретной сцены.

### `output_sections`

Что именно модель должна вернуть.

### `must_preserve`

Что модель не имеет права ломать или игнорировать при генерации.

## `future_extensions`

Список будущих направлений развития. Это не исполняемая логика, а продуктовый backlog внутри схемы.

## Why this structure matters

Главная идея этого каркаса: отделить структуру сценария от литературной подачи.

- JSON отвечает за маршрут, правила, типы сцен и границы.
- LLM отвечает за живой текст, атмосферу, переходы и адаптацию формулировок.
- Такой подход помогает сделать игру управляемой, расширяемой и менее хаотичной.

## Recommended section split

Ниже практическое разделение секций по ролям.

### Engine required

Эти секции нужны самому движку сценария:

- `schema_version`
- `scenario_id`
- `scenario_type`
- `status`
- `session_model`
- `world_setup`
- `progression`
- `question_policy`
- `branching_policy`
- `summary_policy`
- `scene_flow`

### Prompt required

Эти секции или подполя нужно передавать LLM для качественной генерации:

- `product_goal`
- `experience_principles`
- `world_setup`
- `question_policy`
- `branching_policy`
- `summary_policy`
- `scene_flow[].scene_type`
- `scene_flow[].purpose`
- `scene_flow[].ladder_stages`
- `scene_flow[].psychological_goal`
- `scene_flow[].allowed_question_families`
- `scene_flow[].forbidden_question_families`
- `scene_flow[].question_templates`
- `scene_flow[].transition_goal`

### Editorial optional

Эти секции полезны для людей, но не обязательны ни движку, ни промпту в каждом вызове:

- `scenario_name`
- `description`
- `scene_flow[].title`
- `future_extensions`

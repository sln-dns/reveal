# MVP Data Model

Модель покрывает первый runtime-сценарий с двумя участниками и отделяет состояние исполнения от сгенерированного контента.

## Сущности

### `session`

Корневой контейнер пользовательской сессии. Хранит выбранный `scenario_key`, общий lifecycle-статус, агрегированное runtime-состояние в `lifecycle_state` и технические метаданные в `metadata_payload`.

### `session_participant`

Участник сессии. Поле `slot` фиксирует место участника в паре (`1`/`2`), `state_payload` хранит runtime-состояние участника, а `generated_profile` держит сгенерированные артефакты профиля отдельно от state.

### `scenario_run`

Конкретный запуск сценария внутри сессии. Нужен для хранения статуса прохождения, версии сценария, текущей сцены и агрегированного `runtime_state`. Поле `generated_content` предназначено для материалов, которые производит движок или LLM и которые не являются управляющим state.

### `scene_instance`

Runtime-экземпляр сцены внутри `scenario_run`. Хранит порядок прохождения (`position`), статус, state конкретной сцены и generated content сцены отдельно.

### `question_instance`

Экземпляр вопроса, адресованный конкретному участнику. Это позволяет поддерживать асинхронную механику: у каждого участника свой вопрос и свой статус доставки/ответа. `state_payload` хранит runtime-флаги, `prompt_text` и `prompt_payload` содержат пользовательский текст и дополнительные сгенерированные данные.

### `answer`

Фактический ответ участника на `question_instance`. Хранит текст ответа и структурированное представление в `content_payload`, если понадобится нормализация или разметка.

### `summary`

Сводка, сгенерированная по сцене или по всему `scenario_run`. Поле `kind` различает scene-level и run-level summary. Это отдельная сущность, чтобы generated output не смешивался с runtime-state.

## Почему схема минимальна

- Поддерживает core loop: сессия -> участники -> запуск сценария -> сцены -> вопросы -> ответы -> summary.
- Поддерживает двух участников без жёсткой привязки логики к ровно двум строкам в коде: это обеспечивается `slot` и связями.
- Не привязана к SQLite-специфике: используются переносимые типы SQLAlchemy (`String`, `Text`, `JSON`, `DateTime`), поэтому переход на PostgreSQL потребует в основном смены URL и адаптации миграций по мере усложнения схемы.
